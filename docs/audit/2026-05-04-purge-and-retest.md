## Bugs surfaced by clean-slate retest

### 🔴 F1 — Dashboard `/` is statically prerendered with build-time DB counts

**Severity: critical.** The dashboard tiles show stale numbers from when `pnpm build` was last run (S11):

| Tile | Live DB | Dashboard shows |
|---|---:|---:|
| Active runs | 0 | **3** (from old DB) |
| Denials · 24h | 2 | **546** (from old DB) |
| Active pauses | 0 | 0 (coincidentally matches) |

**Root cause:** `apps/web/app/page.tsx` has no `export const dynamic = 'force-dynamic'`. Next.js 15 defaults to static prerender for any route without dynamic deps — and DB queries via `createWebDb()` aren't recognized as dynamic by the compiler. Confirmed via `.next/prerender-manifest.json`:

```json
{
  "routes": ["/_not-found", "/packs", "/settings/team", "/templates", "/settings/account", "/"]
}
```

Static-baked HTML lives at `apps/web/.next/server/app/index.html` — identical bytes returned regardless of cache-busting query string.

**Fix:** add `export const dynamic = 'force-dynamic'` to `app/page.tsx`, `app/packs/page.tsx`, `app/packs/[slug]/page.tsx`, `app/templates/page.tsx`. (`/projects`, `/runs`, `/policies`, `/kill-switches`, `/runs/[id]` are already dynamic — confirmed via no `.html` file in `.next/server/app/`.)

### 🔴 F2 — `__global__` sentinel leaks into `/projects`

**Severity: high.** Confirmed live: `/projects` rendered HTML mentions `__global__` 8 times (twice with the human-readable name "Global Policy Rules") plus `coodra-dev` 2 times. Users see an extra "project" they cannot delete.

**Fix:** in `apps/web/lib/queries/projects.ts::listProjects()`, add a `WHERE slug != '__global__'` filter (or a more general `WHERE id NOT IN (well_known_sentinels)`).

### 🔴 F3 — `run_events` orphan path: events from un-registered cwd land with NULL `run_id`

**Severity: high.** During the retest, **23 of 28 `run_events` rows have `run_id = NULL`**. They are not synthetic — they are from this very Claude Code session running in `/Users/abishaikc/Coodra` (which has a stale `.coodra.json` pointing at a now-deleted `coodra` project). The `tool_input` shows real Bash commands like `node packages/cli/dist/index.js doctor` and `lsof -i :3000` — i.e. my own commands.

**Root cause:** when a session emits hook events but no project / run row exists for the cwd, the run-recorder still persists the event row with `run_id = NULL` instead of either:
- (a) auto-creating a `__global__`-scoped synthetic run, or
- (b) rejecting the event with a structured 4xx, or
- (c) discarding silently and incrementing a `dropped_events` counter.

Owning module: `apps/hooks-bridge/src/lib/run-recorder.ts` (per Phase 2 spec brief).

**Pre-purge data confirms this is not a new bug** — 99.86% of pre-purge events were NULL-`run_id` (1,405 of 1,407), accumulated over weeks of multi-session activity against a partially-registered project pool.

**Fix:** decide policy (a / b / c above) in Phase 2 OQ; implement at run-recorder level. Also add a one-shot data-cleanup migration to backfill NULL `run_id` rows to the `__global__` sentinel run (or DELETE them with a kept-row-count log).

### 🟡 F4 — Dashboard "Latest events" table renders orphan rows with `Run = —` (em-dash)

**Severity: medium UX.** Even after F1+F3 are fixed, orphan events from concurrent sessions will still display in the Latest events table because the dashboard query uses `LIMIT 10 ORDER BY created_at DESC`. They render with an em-dash in the Run column (link to nowhere) and `toolu_*` Anthropic-format `tool_use_id` values that are visually distinguishable from synthetic test ids but otherwise meaningless to a user.

**Fix:** either (a) `WHERE run_id IS NOT NULL` in `fetchLatestEvents()` or (b) keep the query but render orphan events with a labelled "Untracked" badge so they are visibly distinct.

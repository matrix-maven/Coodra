# Module 05 — Agent-Driven NL Assembly — Implementation Plan

> **Status:** kickoff (2026-05-08). No slice has landed yet. This document operationalizes `spec.md` §11 into a concrete sequence of changes with file-level diffs, verification gates, and risk markers.
> **Read alongside:** `spec.md` (this directory) — the philosophy and acceptance criteria; `techstack.md` — the in/out tech inventory.

## 0. Working principles

- **Slice-by-slice merges to `main`.** Each slice ships behind no flag — it's a bounded refactor that leaves the system green at every commit. Tests pass before and after each slice.
- **No flagging the new tools off.** When `list_context_packs` and `read_context_pack` land, they're live for every agent immediately.
- **No deprecation period for embeddings.** `summary_embedding` column has zero rows populated today (no producer ever existed). Dropping it in S2 is genuinely zero-impact.
- **Migrations are immutable once on `main`.** `0010` and `0011` ship in distinct PRs so they can be cherry-picked or rolled back independently if dist tarball regeneration surfaces issues.
- **CI is the gate.** The grep test for "M05 service" / "deferred to Module 05" / `EMBEDDING_DIM` strings is added in S8 — it catches accidental re-introductions on every PR after this module.

## S1 — Schema additions (`0010_m05_agent_driven.sql`)

**Goal:** `decisions` and `context_packs` accept the new metadata. Old code still compiles. Old code still runs (new columns are optional).

### Files

- `packages/db/src/schema/sqlite.ts` — add columns to `contextPacks` (`source`, `meta`) and `decisions` (`context`, `impact`, `confidence`, `reversible`)
- `packages/db/src/schema/postgres.ts` — same shape, dialect-appropriate types (JSONB, BOOLEAN, CHECK constraints)
- `packages/db/drizzle/sqlite/0010_m05_agent_driven.sql` — generated by `pnpm db:generate`
- `packages/db/drizzle/postgres/0010_m05_agent_driven.sql` — generated by `pnpm db:generate`
- `packages/db/__tests__/unit/schema-parity.test.ts` — extend the parity assertion to cover the new columns

### Steps

1. Edit both schema files to add the columns. SQLite uses TEXT for JSON-encoded arrays (consistent with existing `decisions.alternatives`); Postgres uses JSONB.
2. Run `pnpm db:generate`. Verify the produced migration files match the design in `spec.md` §9.1.
3. Review SQL by eye — Drizzle occasionally emits surprising CREATE TABLE rebuilds for SQLite when defaults change. We want pure `ALTER TABLE ADD COLUMN` here.
4. Run `pnpm test:integration`. The `schema-parity.test.ts` should pass with the extended assertions; postgres-migrate.test.ts should apply the migration cleanly to a fresh testcontainers Postgres.
5. Commit. PR title: `feat(05/S1): add agent-driven metadata columns`.

### Verification gate

- `pnpm test:integration packages/db` is green
- A fresh `data.db` file booted from `0000` through `0010` has the new columns
- `summary_embedding` column is still present (S2 removes it)

### Risk markers

- **SQLite ALTER TABLE limitations.** SQLite supports `ADD COLUMN` cleanly; this is the safe direction. Avoid CHECK constraints on SQLite for `source` — enforce at app layer in the handler. Postgres CHECK is fine.
- **Idempotency keys on `decisions`.** Existing rows have `idempotency_key = sha256(description)`. New rows from agents that supply `context`/`impact` will have the same shape. No backfill needed.

  - Manifest: imperative rewrite per `spec.md` §5.4
- `apps/mcp-server/src/tools/record-decision/{schema,handler,manifest}.ts`
  - Schema: add optional `context`, `impact`, `confidence`, `reversible` fields
  - Handler: validate (Zod enum on confidence); persist on insert; **idempotency key unchanged** (per OQ-5)
  - Manifest: imperative rewrite per `spec.md` §5.5
- `apps/mcp-server/src/tools/query-codebase-graph/{schema,handler,manifest}.ts`
  - Schema: add optional `maxNodes: number` (default 1000), add `graph_too_large` soft-failure branch
  - Handler: cap result; remove `query_filtering_deferred_to_m05` notice
  - Manifest: drop "deferred to Module 05" clause from description
- `apps/hooks-bridge/src/lib/auto-context-pack.ts` — remove "Module 05 will replace this" comment (line 93); set `source = 'bridge_auto'` on the auto-save path

### Steps

1. Search-packs-nl rewrite. The handler simplification is the biggest single edit; the schema diff is small.
2. Save-context-pack: the new `source` semantics are the trickiest part. The handler logic:

   ```ts
   const existing = await getByRunId(runId);
   if (existing === null) {
     await insert({ ...payload, source: 'agent' });
     return { ok: true, status: 'created' };
   }
   if (existing.source === 'bridge_auto' && /* this call is from agent */) {
     await update({ ...payload, source: 'agent' });
     return { ok: true, status: 'upgraded_from_bridge_auto' };
   }
   // Otherwise — same agent re-call, or bridge re-call — no-op
   return { ok: true, status: 'noop', existing };
   ```

   The "this call is from agent" check is implicit: any explicit MCP call is by definition agent-side. The bridge calls into the same handler from `auto-context-pack.ts`, but with a flag/header that distinguishes (or via a separate internal entrypoint that always sets `source='bridge_auto'`). Choose the latter — cleaner separation.
3. Record-decision: straightforward column additions. Verify Zod confidence enum maps to dialect-appropriate types (string in SQLite, varchar in Postgres).
4. Query-codebase-graph: cap + soft-failure addition.
5. Auto-context-pack: edit comment + add `source` parameter to the insert.

### Verification gate

- `pnpm typecheck` clean
- `pnpm test:unit` clean — including new tests:
  - `save_context_pack` with no existing row → creates with `source='agent'`
  - `save_context_pack` after bridge auto-save → upgrades, content replaced, source flipped
  - `save_context_pack` agent → bridge → second call no-ops
  - `save_context_pack` agent → agent → second call no-ops
  - `record_decision` with all new fields → row has them populated
  - `record_decision` with new fields, same description, twice → second collapses (idempotency)
  - `search_packs_nl` finds matches in the wider scope (substring inside `content` byte 1500-2000)
- `pnpm test:integration` clean
- Manifest test still asserts 10 tools

### Risk markers

- **The `source` upgrade is the one ADR-007 relaxation in M05.** Document it in the handler comment + in `essentialsforclaude/11-adrs.md` ADR-007 footnote: "Single exception: M05 allows agent-explicit save_context_pack to overwrite when existing source='bridge_auto'."
- **`save_context_pack` already has callers in production.** The new `meta` field is optional; existing callers continue to work. The `source` change is invisible to clients (it's a server-side detail of how the row is persisted).

### Implementation notes

- **Pagination cursor for `list_context_packs`:** opaque base64 of `{lastCreatedAt, lastId}`. Validate on input (return malformed_cursor soft-failure on bad).
- **`read_context_pack` size cap:** content > 200KB → `pack_too_large` with `howToFix: "retry with excerptOnly: true"`. The 500-char excerpt is always cheap.
- **Decisions hydration:** join on `decisions.run_id = context_packs.run_id`, ordered chronologically (`created_at ASC`). Cap at `decisionsLimit` (default 50, max 200).
- **Discriminated union output:** Zod tagged union — `{ found: true, ... }` vs `{ found: false }` vs `{ ok: false, error: 'pack_too_large', ... }`. Agents check `response.data.found` and `response.data.ok` per the standard caller contract.

### Verification gate

- All unit + integration tests pass
- Manifest test green at 12 tools
- Both new manifests pass `assertManifestDescriptionValid` (40-80 words, imperative trigger, return shape mentioned)

### Risk markers

- **Token budget on `read_context_pack`.** A 50-decision project with rich `context` + `impact` JSON could push the response past 100KB. Test with realistic data; tune the default `decisionsLimit` if needed.

### Implementation notes

- **Format** per `spec.md` §7.4. Decisions written before M05 (no `context`, `impact`, `confidence`, `reversible`) render gracefully — empty fields omitted.
- **Confidence-aware ordering** per `spec.md` §7.3 — `ORDER BY (CASE confidence WHEN 'high' THEN 1 ...) , created_at DESC`.
- **Fail-open** — any DB error logs warn and returns `null`. SessionStart proceeds with Feature Pack only.
- **Project resolution** — bridge already resolves `projectId` for SessionStart from `.contextos.json` + `runs` row. Reuse that.

### Verification gate

- Unit tests:
  - Formatter produces expected markdown for fixture rows
  - Empty result → `null` returned (not empty markdown header)
  - DB error → `null` returned, warn logged
  - Confidence ordering applied correctly on truncation
- Integration test:
  - Project with 15 decisions, limit 10, age 30 days → returns 10 decisions, oldest within 30 days
  - Project with 15 decisions, limit 5 → top 5 by (confidence, recency)
  - Project with 0 decisions → `additionalContext` does not include the recent-decisions section
  - Project with `recentDecisionsLimit: 0` → feature disabled, section not rendered
- E2E:
  - SessionStart hook against a real seeded project → `additionalContext` contains all three blocks (Feature Pack, Session Contract, Recent Decisions) in order, separated by `---`

### Risk markers

- **`additionalContext` size.** Feature Pack + Contract + 10 decisions ~= 5-15KB. Agent's context window absorbs this fine, but Claude Code may have its own ceiling on `additionalContext`. Verify with a real session.
- **Privacy.** Recent decisions surface to whoever opens a session. In team mode, this means dev B sees dev A's decisions. That's the point — but worth flagging for security review when team mode lights up.

## Sequencing recommendations

- S1 + S2 + S3 ship in **one PR** — schema changes and the code changes that depend on them must land together, or `main` is broken between merges. They're "three slices" for documentation, not for git history.
- S4 + S5 can ship as separate PRs (existing-tool changes are independent of new-tool addition).
- S6 + S7 can ship in parallel branches; both touch the bridge's SessionStart but in different code paths.
- S8 (documentation) ships last so it reflects the final state.
- S9 (Context Pack save) is the merge-to-main moment, fired as the final commit.
